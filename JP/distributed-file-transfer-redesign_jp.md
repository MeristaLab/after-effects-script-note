# 分散システムのファイル転送方式再設計

## 経緯

After Effectsの分散レンダリングシステムを個人開発中、ファイル転送の実装を進める中でGET（ダウンロード）とPOST（アップロード）の管理方式が異なっている。

```
GET  → Range: bytes=N-
POST → { chunk_index: 2, total_chunks: 10 }
```

それぞれ一般的な手法として採用していたが、なぜこの2つが異なる方式なのかを改めて調査することにした。

## なぜGETとPOSTで方式が異なるのか

GETのRangeはHTTP/1.1で標準化されている。バイト単位で範囲を指定してサーバーに要求する仕組みがプロトコルとして定義されている。

一方POSTには送る側の範囲指定に相当する標準仕様がない。そのため各実装が独自にファイルを分割して順番に送るチャンクアップロードを作るのが一般的になった。非対称になっているのは仕様上の理由からだった。

## 大容量データという前提での問題

チャンクアップロードは実装がシンプルで、軽量なファイルのやり取りには十分機能する。実際よく使われている方式でもある。

ただし今回扱うのは動画ファイルで、数GB〜数十GBになることが前提。その規模になると管理すべきことが増え、中断・再開時のリトライ処理なども自前で組む必要が出てくる。チャンクアップロードはその点で最適ではないという判断になった。大容量ファイルへの対応を前提に、POSTにもより適した方式があるのではと思い調査した。

## TUSプロトコル

大容量ファイルのアップロードに適した方式としてTUSプロトコルが上がってきた。アップロードの中断・再開を標準化したオープンプロトコルで、実装の実績もある。

ただし採用するには問題があった。

TUSのPythonサーバー実装は存在するが、公式リファレンス実装ではなくコミュニティ製のため長期的な安定性に懸念がある。TUSの公式リファレンス実装はGo製のtusdで、これは安定しているが別言語をシステムに追加することになる。

さらにこのシステムはGateway / Controller / Storage / Nodeに分散した構成をとっており、内部通信にまでTUSを使うと管理が重くなる。tusdが持つアップロードIDと自前のjob_idの紐づけも別途必要になる。

チャンクアップロードのままTUSを将来的に追加するルートも考えたが、後から組み込む際の設計変更コストも無視できない。

## Content-Rangeで統一する結論

`Content-Range`というHTTPヘッダーがPOST側にも存在する。

```
GET  → Range: bytes=N-
POST → Content-Range: bytes N-M/Total
```

GETのRangeと対称的に使えてHTTP標準の範囲内。外部依存がなく、分散した内部通信にもそのまま使える。

最初からContent-Rangeで実装しておくことで、デバッグも自分の手で行える。自前実装になる分の工数はかかるが、HTTP標準仕様であるためTUSより汎用性が高く、クライアント側の実装の選択肢も広い。現時点ではこの方式で進める。
